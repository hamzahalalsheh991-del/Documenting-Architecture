# 1. Введение

Целью данной работы является документирование архитектуры программной системы с использованием подхода **Architecture as Code** и языка **Structurizr DSL**.

В рамках задания была разработана архитектура системы **Fitness Tracker**, которая позволяет пользователям вести учёт тренировок и упражнений.

Документация архитектуры построена по модели **C4**, которая включает несколько уровней описания системы.

---

# 2. Описание системы

**Fitness Tracker** — это веб-приложение для управления тренировками.

Система позволяет пользователям:

* создавать упражнения
* формировать тренировки
* просматривать историю тренировок
* анализировать статистику тренировок

Основные сущности системы:

| Сущность     | Описание                      |
| ------------ | ----------------------------- |
| Пользователь | Человек, использующий систему |
| Упражнение   | Физическое упражнение         |
| Тренировка   | Набор упражнений              |

---

# 3. Требования к API

Система должна поддерживать следующий набор API операций.

## Пользователь

* Создание нового пользователя
* Поиск пользователя по логину
* Поиск пользователя по имени и фамилии

## Упражнения

* Создание упражнения
* Получение списка упражнений

## Тренировки

* Создание тренировки
* Добавление упражнения в тренировку
* Получение истории тренировок пользователя
* Получение статистики тренировок за период

# 5. Контекст системы (C1)

Диаграмма **System Context** показывает систему Fitness Tracker во взаимодействии с внешними акторами.

В данной архитектуре внешний актор:

* Пользователь (User)

Пользователь взаимодействует с системой через веб-интерфейс.

---

# 6. Архитектурный подход

Архитектура системы построена на основе **микросервисного подхода**.

Основные принципы:

* разделение сервисов по бизнес-логике
* независимые базы данных
* взаимодействие сервисов через REST API
* использование SPA-интерфейса

Для документирования используется модель **C4**.

| Уровень | Диаграмма         |
| ------- | ----------------- |
| C1      | System Context    |
| C2      | Container Diagram |
| C3      | Component Diagram |
| Dynamic | Runtime сценарий  |

# 7. Контейнерная архитектура (C2)

Система состоит из следующих контейнеров.

| Контейнер               | Технология         | Назначение                 |
| ----------------------- | ------------------ | -------------------------- |
| Single Page Application | React              | Пользовательский интерфейс |
| API Gateway             | Node.js / Express  | Маршрутизация запросов     |
| User Service            | Java / Spring Boot | Управление пользователями  |
| Exercise Service        | Java / Spring Boot | Управление упражнениями    |
| Workout Service         | Java / Spring Boot | Управление тренировками    |
| User Database           | PostgreSQL         | Хранение пользователей     |
| Exercise Database       | PostgreSQL         | Хранение упражнений        |
| Workout Database        | PostgreSQL         | Хранение тренировок        |

---

# 8. Взаимодействие контейнеров

# 11. Динамический сценарий

## Создание тренировки

Рассмотрим основной сценарий работы системы.

### Последовательность действий

1. Пользователь создаёт тренировку через интерфейс SPA.
2. SPA отправляет запрос:

```
POST /api/workouts
```

3. API Gateway принимает запрос.
4. API Gateway передает запрос в **Workout Service**.
5. Workout Service проверяет упражнения через **Exercise Service**.
6. Workout Service проверяет пользователя через **User Service**.
7. Если проверки успешны, тренировка сохраняется в **Workout Database**.
8. Сервис возвращает ответ:

```
201 Created
```

9. SPA отображает пользователю сообщение об успешном создании тренировки.
